# Все сценарии чеков

## 1. Оплата / приход

Обычный чек аванса по заказу покупателя.

- операция: `sell`;
- канал: API АТОЛ;
- наличные или безналичные по выбору;
- товар: `payment_object = payment`, метод `advance`;
- услуга: `full_prepayment`, агент и поставщик передаются в позиции;
- товарная ставка: `vat122`;
- агентская услуга: ставка из настройки агента;
- собственная доставка по России: без агента.

## 2. Возврат по заказу

Фактический возврат денег покупателю по сообщению бухгалтера.

- операция: `sell_refund`;
- канал: API АТОЛ;
- способ оплаты выбирается отдельно: наличные или карта;
- ФИО берётся из сообщения, а при отсутствии может быть загружено из контактного
  лица заказа покупателя в 1С;
- обычный товар: `vat122`;
- услуга: ставка и агент определяются по услуге и подразделению;
- тег 1192 не используется, потому что это реальный возврат, а не исправление.

## 3. Обычная реализация

Обычные реализации должны пробивать кассиры в 1С. В программе сохранён редкий
ручной режим:

- операция: `sell`;
- канал: API АТОЛ;
- `full_payment`;
- товарная ставка `vat22`;
- для услуги передаются номенклатура, агент и поставщик.

## 4. Реализация без чека

Реализация существует, обычный чек в дату расчёта не был сформирован.

- если дата прошла: `sell_correction`;
- канал: XML и ручная загрузка в АТОЛ;
- API в интерфейсе заблокирован;
- табличная часть не нужна для используемого XML ФФД 1.05;
- ставка определяется по товару/услуге и подразделению;
- тег 1192 отсутствует, потому что исходного чека нет;
- создаётся одна служебная записка.

## 5. Реализация с ошибочным чеком

Исходный чек найден по ФПД, но пробит неверно или не той датой.

- обычный обратный чек, обычно `sell_refund`;
- затем правильный обычный `sell`;
- канал: XML;
- тег 1192 присутствует в обоих чеках;
- исходный и правильный чеки могут иметь разные суммы, оплату и номенклатуру;
- одна служебная записка описывает оба чека.

## 6. Obsidian-исправления

Сценарии определяются по пояснению, данным 1С и исходному чеку ОФД. Это отдельный
процесс и отдельное окно. Возможны полная отмена, исправление суммы, оплаты,
номенклатуры, даты, отсутствующего чека и расходной операции.

Все Obsidian-исправления XML-only. Реальные возвраты из этого раздела исключены.
Подробности: [obsidian-corrections.md](obsidian-corrections.md).

## 7. Коррекция расхода

- без исходного чека: один `buy_correction`;
- при существующем ошибочном чеке: обычная обратная операция и правильный
  обычный чек с тегом 1192 в обоих;
- канал: XML;
- сценарий требует ручной проверки, если исходный документ сам является чеком
  коррекции.
