# Исправления из Obsidian

## Назначение

Рабочий Markdown-файл содержит документы, подразделение и краткое пояснение
расхождения между 1С и фискальными данными. Путь задаётся в настройках.

- `[ ]` - случай ещё открыт;
- `[x]` - случай полностью закрыт: исправление выполнено, служебная записка
  проверена, пояснение внесено в 1С.

Старые отмеченные строки считаются подтверждёнными и повторной проверки не
требуют.

## Периоды

Месяц и год берутся из заголовков исходного Markdown. Фильтр `Все периоды`
показывает общую очередь. Смена фильтра не переносит строки между месяцами и не
меняет файл.

## Рабочий порядок

1. Нажать `Обновить`, чтобы перечитать Markdown.
2. Выбрать нужные строки.
3. Нажать `Загрузить данные 1С`.
4. Загрузить отчёт ОФД или нажать `Найти чеки по ФП`.
5. Проверить автоматически предложенный сценарий.
6. Исправить сумму, оплату, услугу или номенклатуру в редакторе при необходимости.
7. Передать готовые строки в отдельное окно пробития исправлений.
8. Сформировать XML и служебные записки.
9. Загрузить XML в АТОЛ Online.
10. Подтвердить чеки по журналу АТОЛ/ОФД, проверить служебные записки и запись в 1С.
11. Закрыть случай; синхронизация поставит `[x]` в Obsidian.

## Что означает план

План - рассчитанный набор будущих чеков. Для каждого чека он показывает:

- порядок;
- операцию;
- сумму;
- способ оплаты;
- ставку НДС;
- необходимость номенклатуры;
- использование ФП исходного чека;
- готовность к XML.

`Нужен сценарий` означает, что пояснения недостаточно для безопасного решения.
Сценарий выбирается вручную в карточке/редакторе.

## Отдельное окно

Кнопка передачи не помещает Obsidian-кейс в обычную форму оплаты. Открывается
самостоятельное окно `Пробитие исправлений`, которое можно держать рядом с 1С и
отчётом ОФД.

В окне нет кнопки `Пробить в АТОЛ Online`. Генерация создаёт только XML и DOCX.
Возврат к списку закрывает окно и возвращает фокус к Obsidian.

## Данные справа

Карточка показывает:

- номер и тип документа;
- дату документа и правильную сумму;
- подразделение;
- номер чека ККМ, ФП и дату печати из 1С;
- комментарий 1С;
- исходный чек ОФД;
- пояснение Obsidian;
- сценарий и готовность к закрытию.

Данные 1С сохраняются в модели случая и не должны исчезать при переключении строк.

## Синхронизация пояснений

Изменённое пояснение записывается обратно в ту же строку Markdown. Изменения
файла, сделанные в Obsidian, появляются после обновления программы. Перед записью
создаётся резервная копия средствами сервиса синхронизации.

## Закрытие

Для нового случая проверяются три независимых условия:

- чек подтверждён в АТОЛ/ОФД;
- служебная записка проверена;
- пояснение внесено в 1С.

Проверка служебных записок может быть ручной или автоматической по настройке.
