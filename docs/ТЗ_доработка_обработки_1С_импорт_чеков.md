# Техническое задание

## Доработка обработки 1С для импорта фискальных данных из AtolGenerator

### 1. Назначение

Доработать внешнюю обработку 1С `ИмпортЧекНомерФПИзCSV`, чтобы она импортировала новый CSV-файл, сформированный программой AtolGenerator после сопоставления:

- исходного XML-файла чеков;
- отчёта из журнала чеков АТОЛ Online;
- при наличии - отчёта ОФД/Такскома для дополнительной проверки.

Прямая запись из AtolGenerator в 1С через COM не используется. Все изменения документов выполняются пользователем интерактивного сеанса 1С через внешнюю обработку.

### 2. Сценарии импорта

#### 2.1. Одиночный чек коррекции

Применяется, когда исходный чек по реализации не был пробит и позднее сформирован один `sell_correction`.

Обработка должна:

1. Записать в документ `РеализацияТоваровУслуг`:
   - `ЧекНомерФП` - ФПД чека коррекции;
   - `НомерЧекаККМ` - номер ФД чека коррекции;
   - `ДатаПечатиЧека` - фактическую дату и время регистрации чека коррекции в АТОЛ.
2. Дописать в `Комментарий` текст вида:

```text
12.07.2026 Пробит чек коррекции "Приход" ФП: 9876543210
```

#### 2.2. Исправительная пара чеков

Применяется, когда неправильный чек уже был записан в реализации, а для исправления сформированы два новых чека:

1. `sell_refund` - исправительный чек `Возврат прихода`;
2. `sell_correction` - чек коррекции `Приход`.

В этом сценарии обработка не должна изменять следующие существующие реквизиты реализации:

- `ЧекНомерФП`;
- `НомерЧекаККМ`;
- `ДатаПечатиЧека`.

В этих реквизитах должны остаться данные первоначального неправильного чека.

Обработка должна только дополнить `Комментарий` сведениями об обоих новых чеках:

```text
12.07.2026 Пробит исправительный чек "Возврат прихода" ФП: 1234567890   ///   12.07.2026 Пробит чек коррекции "Приход" ФП: 9876543210
```

Даты в комментарии - фактические даты регистрации соответствующих чеков в АТОЛ.

### 3. Формат входного CSV

Кодировка: Windows-1251 без BOM.

Разделитель: точка с запятой (`;`).

Первая строка содержит заголовки. Колонки необходимо искать по заголовкам, а не по фиксированным индексам.

```text
НомерРеализации;ТипЧека;РежимЗаписи;ExternalId;ФПД;НомерФД;ДатаЧека;Комментарий
```

Описание колонок:

| Колонка | Назначение |
|---|---|
| `НомерРеализации` | Номер документа `РеализацияТоваровУслуг` без префикса `Основание_` |
| `ТипЧека` | Машинный тип операции: `sell_correction` или `sell_refund` |
| `РежимЗаписи` | `update_fields` или `comment_only` |
| `ExternalId` | Уникальный идентификатор чека из XML и журнала АТОЛ |
| `ФПД` | Фискальный признак документа |
| `НомерФД` | Фискальный номер документа |
| `ДатаЧека` | Фактическая дата регистрации в формате `dd.MM.yyyy HH:mm:ss` |
| `Комментарий` | Готовый текст, который требуется добавить в комментарий реализации |

Значения `РежимЗаписи`:

- `update_fields` - заполнить фискальные реквизиты и добавить комментарий;
- `comment_only` - фискальные реквизиты не изменять, добавить только комментарий.

### 4. Примеры CSV

#### 4.1. Одиночная коррекция

```text
НомерРеализации;ТипЧека;РежимЗаписи;ExternalId;ФПД;НомерФД;ДатаЧека;Комментарий
т0000051448;sell_correction;update_fields;0f1e2d3c4b5a;9876543210;14801;12.07.2026 14:35:00;12.07.2026 Пробит чек коррекции "Приход" ФП: 9876543210
```

#### 4.2. Исправительная пара

```text
НомерРеализации;ТипЧека;РежимЗаписи;ExternalId;ФПД;НомерФД;ДатаЧека;Комментарий
т0000051448;sell_refund;comment_only;a11111111111;1234567890;14802;12.07.2026 15:10:00;12.07.2026 Пробит исправительный чек "Возврат прихода" ФП: 1234567890
т0000051448;sell_correction;comment_only;b22222222222;9876543210;14803;12.07.2026 15:11:00;12.07.2026 Пробит чек коррекции "Приход" ФП: 9876543210
```

### 5. Поиск реализации

Для каждой строки необходимо найти документ `РеализацияТоваровУслуг`:

1. `Док.Номер = НомерРеализации`;
2. `Док.ПометкаУдаления = ЛОЖЬ`;
3. `Док.Дата <= КонецДня(ДатаЧека)`;
4. если номер встречается в нескольких периодах, выбрать самый свежий документ по `Док.Дата УБЫВ`.

Если документ не найден, строка записывается в ошибки, остальные строки продолжают обрабатываться.

### 6. Правила изменения документа

#### 6.1. Режим `update_fields`

1. Получить объект найденной реализации.
2. Если фискальные реквизиты пустые, заполнить их данными строки CSV.
3. Если `ЧекНомерФП` уже заполнен, по умолчанию не перезаписывать документ и вывести конфликт.
4. Перезапись заполненных реквизитов допускается только при явно установленном пользователем флаге `Перезаписывать заполненные`.
5. Добавить комментарий из CSV, если в существующем комментарии ещё нет указанного ФПД.
6. Записать документ.

#### 6.2. Режим `comment_only`

1. Не проверять заполненность `ЧекНомерФП` как причину пропуска строки.
2. Не изменять `ЧекНомерФП`, `НомерЧекаККМ` и `ДатаПечатиЧека` даже при установленном флаге `Перезаписывать заполненные`.
3. Добавить комментарий из CSV, если в существующем комментарии ещё нет указанного ФПД.
4. Если для одной реализации переданы `sell_refund` и `sell_correction`, комментарии добавить в порядке:
   - сначала `sell_refund`;
   - затем `sell_correction`.
5. Между новыми частями комментария использовать разделитель `   ///   `.
6. Если у документа уже есть другой комментарий, сохранить его и дописать новые части через тот же разделитель.
7. Записать документ один раз после формирования полного комментария для реализации.

### 7. Защита от повторного импорта

Повторная загрузка того же CSV не должна:

- повторно добавлять одинаковый комментарий;
- повторно изменять фискальные реквизиты теми же значениями;
- создавать дубли сведений об исправительной паре.

Уникальность проверять минимум по ФПД. `ExternalId` также можно использовать для диагностики и журнала обработки.

### 8. Ограничения

В эту обработку не включать:

- `buy_correction` и чеки коррекции по возвратам;
- обычные чеки `sell`, пробитые через API;
- самостоятельные возвраты, не относящиеся к исправлению реализации;
- изменение реквизита `Ответственный`;
- прямое подключение к АТОЛ или ОФД.

Если в CSV встретился неподдерживаемый `ТипЧека` или неизвестный `РежимЗаписи`, строку пропустить с явным сообщением об ошибке.

### 9. Интерфейс обработки

Сохранить существующие элементы:

- путь к CSV-файлу;
- флаг `Перезаписывать заполненные`;
- кнопку выполнения импорта.

В итоговом сообщении отдельно вывести:

- реализаций с заполненными реквизитами;
- реализаций, где добавлен только комментарий;
- пропущенных дублей;
- конфликтов заполненных реквизитов;
- не найденных документов;
- неподдерживаемых строк;
- ошибок записи.

Флаг `Перезаписывать заполненные` действует только для строк `update_fields` и никогда не разрешает менять фискальные реквизиты в режиме `comment_only`.

### 10. Автор изменений в 1С

Документы записываются из интерактивного сеанса 1С. В истории и журнале регистрации автором должен быть текущий пользователь, запустивший обработку.

Реквизит документа `Ответственный` обработка не изменяет.

### 11. Приёмочные проверки

1. Одиночный `sell_correction` с пустыми реквизитами заполняет три фискальных поля и комментарий.
2. Исправительная пара при уже заполненных реквизитах не изменяет ни одно из трёх полей и добавляет два комментария с разными ФПД.
3. Повторный импорт того же файла не создаёт дубли.
4. Существующий произвольный комментарий документа сохраняется.
5. Флаг перезаписи не влияет на строки `comment_only`.
6. Документ с повторяющимся номером выбирается с учётом даты чека.
7. Ошибка одной строки не останавливает импорт остальных строк.
8. `buy_correction` и другие неподдерживаемые типы не изменяют документы.

### 12. Логика формирования CSV в AtolGenerator

AtolGenerator формирует CSV после следующего сопоставления:

1. XML предоставляет `ExternalId`, номер реализации, тип операции и принадлежность чека к одиночной коррекции или исправительной паре.
2. Отчёт АТОЛ Online предоставляет по `ExternalId` ФПД, номер ФД и фактическую дату регистрации.
3. Отчёт ОФД/Такскома при наличии используется для дополнительной проверки ФПД, ФД, даты, суммы и типа операции.
4. Для одиночного `sell_correction` программа устанавливает `РежимЗаписи = update_fields`.
5. Для обоих чеков исправительной пары программа устанавливает `РежимЗаписи = comment_only`.
6. Номер вида `Основание_т0000051448` нормализуется до `т0000051448` до записи CSV.
